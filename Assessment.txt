Background
You have been handed access to a public GraphQL API. Your mission is to design a targeted test suite that validates the API's core behaviors, handles real-world edge cases, and demonstrates how you use AI tools to accelerate quality engineering work.
We are not looking for exhaustive coverage — we want to see how you think about risk, how you structure a strategy under time constraints, and how you actually use AI in your workflow.
The API: Countries GraphQL API
Use the public Countries GraphQL API:
https://countries.trevorblades.com
(Backup API just in case - https://rickandmortyapi.com/graphql)
You can explore the schema interactively in a GraphQL playground (e.g., Apollo Sandbox, Altair, or Insomnia). No authentication is required.
The API exposes three top-level query types:
•countries — data for every country, including name, capital, currency, phone codes,emoji, and AWS region
•continents — continent codes and names, with nested country relationships
•languages — language codes, names, and native names
All three query types support an optional filter argument. Countries can be filtered by code, currency, or continent code. Continents and languages can be filtered by code. The API supports the following filter operators: eq, ne, in, nin, and regex.
Notable schema details worth exploring:
•Country.currency and Country.phone may return comma-separated values for countrieswith multiple currencies or phone codes
•Country.currencies and Country.phones return proper arrays — useful for data shapetesting
• Country.states returns an array of provinces/states, which may be empty for some countries
• Country.awsRegion returns the nearest AWS region for each country
Part 1: Test Strategy Document
Before writing a single line of code, write a brief test strategy document. This does not need to be long — clarity and reasoning matter more than length.
What to include:
• What questions did you ask (or would ask) before starting to test this API?
• Which 3–5 scenarios did you prioritize, and why?
• What did you consciously choose NOT to test given the time constraint, and why?
• What risks or edge cases concern you most about a GraphQL API of this type?
• How did you use AI tools during the strategy phase — what did you prompt, what did you get, and what did you change?
Format: Any format is fine (Google Doc, Markdown, plain text). Email it to lyle.smart@continued.com before your interview.
Part 2: Automated Test Suite
Write an automated test suite targeting the SpaceX GraphQL API. Use whatever language, framework, or tooling you prefer — what matters is that it runs and demonstrates real API validation.
Your suite must include:
• Required: At least one schema validation test
• Required: At least one query test covering a happy-path scenario
• Required: At least one test for an edge case or error condition (e.g., invalid arguments, null handling, deeply nested fields)
• Required: At least one test that validates data shape or type correctness beyond just a 200 OK response
Bonus (not required, but impressive):
•A test using filter operators (e.g., in, nin, regex) and validating the filtered results arecorrect
•A performance or latency assertion on a query that returns all countries with nestedfields
•A test that validates the consistency between Country.currency (comma-separatedstring) and Country.currencies (array) for a multi-currency country
AI tool requirement:
You must use at least one AI tool (e.g., Claude, GitHub Copilot, Cursor, ChatGPT) during test creation. During the interview walkthrough, be prepared to:
•Show the prompts or interactions you used
•Explain what the AI got right, what it got wrong, and what you had to fix or change
•Describe how it changed (or did not change) your workflow
There is no single right answer here. We want to see your judgment, not just passing tests.
